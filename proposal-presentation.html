1. Project Overview

What problem are we solving?

The Problem

Containers are everywhere, but visibility is lacking

Organizations deploy hundreds of containers with no unified view of what happens inside them at runtime.

Security Blind Spots

  • Unauthorized network connections
  • Unexpected open ports
  • Sensitive file access (/etc/shadow)
  • Resource anomalies (e.g., crypto-mining)
  • Vulnerable base images in production

Why It Matters

  • Containers share the host kernel => one compromise can escalate
  • Short-lived containers make forensics hard
  • Compliance (CIS Benchmarks) requires continuous monitoring

Our Solution

A distributed monitoring platform with three core components:

Lightweight Agent

Deployed per container. Collects:

Network I/O Open Ports File Access CPU / Mem Processes

Central Server (FastAPI)

Aggregates telemetry & enforces security:

Compliance Alerts Vuln Scanner Anomaly Detection

Web Dashboard + Grafana

Real-time operator UI:

Live Metrics Alert Feed CVE Reports Compliance

Similar to: Sysdig Secure, Falco, Datadog Container Monitoring

2. System Model & Assumptions

What kind of distributed system?

System Model

Architecture: Centralized (Hub & Spoke)

Single control-plane server + N distributed agents, one per container. Agents are stateless collectors that push telemetry to the server.

Network: Reliable (Docker bridge/overlay)

Communication over TCP (HTTP + WebSocket). Retries with exponential backoff handle transient failures.

Failure Model: Crash-Failure

No Byzantine faults. Agents/server may crash but don't act maliciously.
Agent crash => server detects via missed heartbeats.
Server crash => agents buffer locally & replay on reconnect.

Consistency Model

Eventual consistency for metrics (1-5s lag OK).
Strong consistency for compliance state & alert acks (PostgreSQL ACID).

4. Algorithms & Protocols

What makes this distributed?

Algorithms & Protocols

Anomaly Detection

Sliding Window + Z-Score
• 5-min rolling average per metric per container
• Alert when current value > 2 std deviations from mean
• Configurable per metric (CPU, memory, net I/O)

Failure Handling

Agent => Server: exponential backoff (1s, 2s, 4s... max 30s)
Heartbeat: 30s missed => "unreachable", 5min => "offline"
Idempotent ingestion: unique batch IDs, server deduplicates

Network Security Rules

• Allowlist / Blocklist for outbound destinations
• Port exposure detection for unexpected listeners
• Connection rate limiting (N conns/sec threshold)

Compliance Rule Engine

JSON predicates evaluated against container state:

{ "condition": "ALL proc.user != root" }

Evaluated periodically + on matching telemetry events

5. Concurrency / Parallel /
Asynchronous / Distributed

How do we handle concurrent operations?

Agent Concurrency (Single Node)

Core: asyncio event loop drives all collectors concurrently

Async Tasks (non-blocking)

  • ResourceCollector => reads /proc + cgroups
  • PortCollector => parses /proc/net/tcp
  • ProcessCollector => psutil.process_iter()
  • Batch sender => drains queue every 5s

Background Threads (blocking I/O)

  • FileCollector => watchdog (inotify) thread
  • NetworkCollector => raw socket recv() thread

Synchronization

  • asyncio.Lock => shared telemetry buffer
  • asyncio.Queue => thread-to-async bridge

Server & System-Level Concurrency

Server (Single Node)

  • FastAPI + Uvicorn => async ASGI request handling
  • Gunicorn => multiple worker processes (1 per core)
  • asyncio.create_task => background jobs (compliance, scans)
  • SQLAlchemy async pool => connection management
  • Redis Pub/Sub => decouples ingestion from alerting & WebSocket

Distributed Properties

  • Agents are independent => no inter-agent communication
  • Natural load distribution => each agent handles one container
  • Server is stateless => all state in PostgreSQL + Redis
  • Horizontally scalable => add server instances behind a load balancer
  • Self-scheduling => agents manage their own intervals

8. Challenges & Difficulties

Challenges & Mitigations

ChallengeMitigation
Race conditions in shared telemetry buffer asyncio.Lock + asyncio.Queue for thread-to-async bridge
Agent overhead on monitored container Read /proc directly instead of spawning commands; configurable intervals
Data volume at scale 5s batching, server-side sampling, PostgreSQL time partitioning
Clock skew across agents UTC everywhere; server applies receipt timestamp for ordering
Network capture needs privileges Agent runs with NET_RAW + SYS_PTRACE capabilities
Trivy scan blocking the API Async subprocess; results streamed via WebSocket
WebSocket fan-out at scale Redis Pub/Sub as broadcast layer, one coroutine per client
